1. التدفق الحالي — ازاي الـ Request بيتمشى

لما العميل يعمل طلب (مثلاً POST /login)، الكود الحالي بيتمشى بالترتيب ده:

١. HTTP Request (POST /api/v1/auth/login)
⬇
٢. Express Server (server.ts)
express.json() middleware
⬇
٣. Router (routers/user.router.ts)
يطابق المسار /api/v1/auth/login
⬇
٤. Validation Middleware
Zod — يتحقق من شكل البيانات
⬇
٥. Controller (controllers/user.controllers.ts)
يستقبل req, ينادي User.class
⬇
٦. User.class (classes/User.class.ts)
Business Logic — login()
⬇
٧. Prisma ORM (utils/connectDB.ts)
Queries قاعدة البيانات
⬇
٨. User.class يرجع الـ data
⬇
٩. Controller يضبط الـ cookies ويرد Response
⬇
١٠. HTTP Response (JSON + Cookies)
ملاحظة: مفيش isAuthenticated middleware في login لأنه route عام. لكن routes protected زي logout بتضيف خطوة إضافية بعد الـ Zod وقبل الـ Controller.

4. إيه اللي غلط في الهيكل الحالي؟

#المشكلةالتفصيل
1 ❌ Service بترجع Middleware User.class.refreshAccessToken() مبتنفذش الحالة — بترجع function محتاجة (req, res, next).
ده معناه إن الـ Service بيعتمد على Express objects (req/res)، وده غلط لأن Service المفروض يبقى pure business logic.
2 ❌ ملفات مبعثرة مش موديولات دلوقتي عندنا controllers/, classes/, validators/ كلها منفصلة. كل موديول مفروض يكون في فولدر واحد.
- classes/User.class.ts فيه Auth + Brand logic مع بعض
- controllers/user.controllers.ts فيه كل Auth controllers
لو عايز تضيف موديول جديد، هتعمل فولدر جديد ولا هتزود في الملفات القديمة؟
3 ❌ مفيش Data Layer (Repository) كل queries الـ Prisma جوا الـ Service مباشرة — prisma.user.findFirst(), prisma.user.update() إلخ.
في الشركات، بيفصلوا queries في auth.model.ts أو auth.repository.ts عشان:
- Testing أسهل (تقدر تستخدم mock للـ repository)
- لو غيرت ORM مستقبلًا (Prisma → Drizzle) التغيير في مكان واحد
4 ❌ Response Logic في Service دوال logout(), verifyRegisterOtp(), verifyOtp(), resetPassword() كلها بتستدعي res.clearCookie() جوا الـ Class method.
الـ Service مش المفروض يعرف حاجة عن الـ Response — ده شغل الـ Controller.
5 ❌ دوال static في الـ Class User.uploadPicture() و User.deletePicture() static methods — ده مش design pattern واضح. المفروض يكون في separate service للـ media أو utility.
6 ❌ await jwt.verify await jwt.verify(...) — jwt.verify مش async أصلاً. مش غلط فادح لكن بيضلل.

5. مثال عملي: Login — كما هو حالياً

الـ Controller بينادي الـ Class والـ Class بيعمل كل حاجة:

📄 controllers/user.controllers.ts
// Controller بسيط — بس بينادي الـ class export const login = asyncHandler( async (req, res) => { const { identifier, password } = req.body; // ← استدعاء Class مباشرة const user = await new User().login(identifier, password); // ← Controller بيضبط cookies res.cookie("accessToken", user.data.accessToken, {...}); res.cookie("refreshToken", user.data.refreshToken, {...}); return res.json(new ApiResponse(...)); } );
📄 classes/User.class.ts — login()
// الـ Class بيعمل كل حاجة: // 1. البحث في DB // 2. التحقق من password // 3. lockout logic // 4. توليد tokens // 5. تحديث DB // 6. إرجاع data async login(identifier, password) { const user = await prisma.user.findFirst({...}); if (user.lockedUntil > new Date()) throw new ApiError("Account locked", 403); const valid = await bcrypt.compare(password, user.password); if (!valid) { // تحديث failedLoginAttempts await prisma.user.update({...}); throw new ApiError("Invalid credentials", 401); } const accessToken = generateAccessToken({ id: user.id }); const refreshToken = generateRefreshToken({ id: user.id }); await prisma.user.update({ where: { id: user.id }, data: { failedLoginAttempts: 0, lockedUntil: null, refreshToken } }); return { success: true, data: { accessToken, refreshToken, ... } }; }
🔍 ملاحظة: طريقة login() مش من أسوأ المشاكل — بترجع data عادي ومش بتستدعي res.cookie. المشكلة الأكبر في دوال تانية (logout, verifyOtp, refreshAccessToken) اللي بترجع middleware بتاخد req, res, next.

6. مثال عملي: Login — بعد التحويل (Standard)

📄 modules/auth/auth.controller.ts — NEW
// Controller بيستدعي Service ويرد Response export const login = asyncHandler( async (req, res) => { const { identifier, password } = req.body; // ← استدعاء Service — بياخد data ويرجع data const result = await authService.login(identifier, password); // ← الـ Controller هو اللي يتعامل مع cookies res.cookie("accessToken", result.accessToken, {...}); res.cookie("refreshToken", result.refreshToken, {...}); return res.json({ success: true, data: result }); } );
📄 modules/auth/auth.service.ts — NEW
// Service بيركز على Business Logic بس class AuthService { async login(identifier: string, password: string) { // ← Data Layer (Model/Repository) const user = await authModel.findUserByIdentifier(identifier); if (user.lockedUntil > new Date()) throw new AppError("Account locked", 403); const valid = await bcrypt.compare(password, user.password); if (!valid) { await authModel.incrementFailedAttempts(user.id); throw new AppError("Invalid credentials", 401); } const tokens = tokenService.generateTokens({ id: user.id }); await authModel.updateAfterLogin(user.id, tokens.refreshToken); return { ...tokens, user: sanitizeUser(user) }; } }
📄 modules/auth/auth.model.ts — NEW
// Data Layer — بس Queries قاعدة البيانات class AuthModel { async findUserByIdentifier(identifier: string) { return prisma.user.findFirst({ where: { OR: [{ email: identifier }, { phone: identifier }] } }); } async incrementFailedAttempts(userId: string) { return prisma.user.update({...}); } async updateAfterLogin(userId: string, refreshToken: string) { return prisma.user.update({...}); } }
📄 modules/auth/auth.validators.ts — NEW
// حاطط Zod schemas جنب الموديول مش بعيد import { z } from "zod"; export const loginSchema = z.object({ identifier: z.string().min(1), password: z.string().min(6), });
✅ خلينا نشوف الفرق:
قبل: Controller ينادي Class → Class فيه Prisma queries + business logic + أحيانًا response logic
بعد: Controller → Service (business logic) → Model (queries)
كل طبقة ليها مسؤولية واحدة واضحة. Testing سهل — تقدر تختبر Service بدون Express، وتختبر Model بدون business logic.

8. خطة التنفيذ — خطوة بخطوة

1

إنشاء الهيكل الجديد (First Commit — لا تغيير في الكود)

ننشئ الفولدرات الجديدة: src/modules/auth/, src/modules/brand/, src/shared/, src/config/
ننقل الملفات كما هي (Copy) بدون تغيير سطر واحد. نتأكد إن المشروع لسه شغال.

2

إصلاح الـ Service Layer — دوال "ترجع middleware"

أهم خطوة. نعدل الدوال دي في auth.service.ts عشان ترجع data مش middleware:

// قبل — بترجع middleware
async refreshAccessToken() {
  return async (req, res, next) => {
    // logic هنا...
    return { accessToken, refreshToken };
  };
}

// بعد — بتاخد token وترجع data
async refreshAccessToken(incomingRefreshToken: string) {
  // logic هنا
  return { accessToken, refreshToken };
}
3

فصل Response Logic من الـ Service

أي حاجة بتستدعي res.cookie() أو res.clearCookie() جوا الـ Class ننقلها للـ Controller.

4

إنشاء Data Layer (Model/Repository)

ننقل كل استدعاءات Prisma من الـ Service إلى auth.model.ts:

// Service بقى بينادي Model
const user = await authModel.findById(id);
await authModel.updateRefreshToken(userId, newToken);
5

إضافة المفقودات في server.ts

نضيف الـ middleware الناقصة: cookie-parser, cors, helmet, morgan, express-rate-limit

6

مسح الفولدرات القديمة

بعد ما نتأكد إن كل حاجة شغالة، نمسح src/classes/, src/controllers/, src/routers/, src/validators/ القديمة.

7

اختبار كل حاجة

npm run dev — نتأكد إن السيرفر يشتغل وكل الـ endpoints ترجع صح.
