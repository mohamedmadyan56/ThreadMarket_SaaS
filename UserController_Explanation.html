1. نظرة عامة على الـ Controller

ملف src/controllers/user.controllers.ts هو المسؤول عن استقبال طلبات HTTP المتعلقة بالمستخدمين (Auth) وتحويلها إلى دوال في طبقة الـ Business Logic (ملف src/classes/User.class.ts).

الـ Controller مش بيعمل أي logic بنفسه — هو مجرد "وسيط" بياخد الـ request، وينادي الدالة المناسبة، ويرجع الـ response.

3. تتبع التدفق الكامل لـ Refresh Token

لما الـ access token يخلص (بتاع 24 ساعة بنسبة للمشروع ده)، بدل ما العميل يسجل دخول تاني، بنستخدم الـ refresh token عشان نجيب واحد جديد.

العميل يطلب تجديد التوكنات
⬇
الـ Controller يستقبل الطلب وينشئ User object
⬇
ينادي user.refreshAccessToken() → ترجع middleware function
⬇
الـ Controller ينفذ middleware function بـ (req, res, next)
⬇
يستخرج refreshToken من الـ cookies أو الـ Authorization header
⬇
يتأكد من صحة التوكن (jwt.verify) باستخدام REFRESH_TOKEN_SECRET
⬇
يلاقي المستخدم في قاعدة البيانات و يقارن التوكن
⬇
يتأكد من إن الحساب مش banned و مش locked
⬇
يولد accessToken + refreshToken جدد
⬇
يحفظ الـ refreshToken الجديد في DB (يلغي القديم)
⬇
الـ Controller يضبط الـ cookies بالتوكنات الجديدة
⬇
يرجع Response للعميل
⚠️ مهم: الـ refresh token بيتم تدويره (Rotation) - كل مرة نعمل refresh، بنولد refresh token جديد وبنلغي القديم. ده معناه إن لو حد سرق الـ refresh token، أول ما الضحية تعمل refresh، السارق بيقعد معاه توكن ملغي.

4. شرح الـ Controller: refreshAccessToken

من ملف src/controllers/user.controllers.ts — الأسطر 142 إلى 161

// === السطر 142: تصدير الدالة ===
export const refreshAccessToken = asyncHandler(
  async (req: Request, res: Response, next: NextFunction) => {

    // === السطر 144: إنشاء instance من User class ===
    const user = new User();

    // === السطر 145: استدعاء business logic ===
    //     الدالة refreshAccessToken() في User.class.ts ترجع MIDDLEWARE
    //     يعني هي دالة جوه دالة (Closure) محتاجة (req, res, next)
    const refreshMiddleware = await user.refreshAccessToken();

    // === السطر 146: تنفيذ الـ middleware ===
    //     بنادي الـ middleware function وندهيلها req, res, next
    //     الدالة جواها كل logic التجديد وترجع result object
    const result = await refreshMiddleware(req, res, next);

    // === السطر 148-155: ضبط الـ cookies بالتوكنات الجديدة ===
    //     بنحط accessToken في cookie مع срок الصلاحية بتاعه
    res.cookie("accessToken", result.data.accessToken, {
      ...baseCookieOptions,
      maxAge: result.data.accessTokenExpiration * 1000 * 60,
    });
    //     بنحط refreshToken في cookie مع срок الصلاحية بتاعه
    res.cookie("refreshToken", result.data.refreshToken, {
      ...baseCookieOptions,
      maxAge: result.data.refreshTokenExpiration * 1000 * 60,
    });

    // === السطر 157-160: إرجاع الـ Response ===
    return res.status(StatusCodes.OK)
      .json(new ApiResponse(result.success, result.message, result.data));
  },
);

شرح السطر سطر:

السطرالشرح
142export const refreshAccessToken - بنصدر الدالة عشان نستخدمها في الـ Router. asyncHandler هو wrapper بيلف الدوال الـ async عشان يمسك الأخطاء ويورّيها للـ error handler بدل ما نعمل try/catch في كل دالة.
143الدالة بتاخد req (الطلب), res (الرد), next (عشان نمرر للخطوة التالية).
144new User() - بننشيء instance من الكلاس عشان نستخدم دواله.
145user.refreshAccessToken() - بنادي الدالة في User.class.ts. الدالة دي مبتنفذش الحالة فورًا — هي بتجهز middleware function وترجعها. refreshMiddleware هو pointer جاهز للتنفيذ.
146refreshMiddleware(req, res, next) - بننفذ الـ middleware function ونديها req, res, next. جواها بيحصل التحقق والتجديد. النتيجة (result) فيها الـ tokens الجدد.
148-151بنضبط cookie اسمه accessToken بالقيمة الجديدة، مع خيارات httpOnly: true (مش accessible من JavaScript)، secure في production، sameSite مناسب. maxAge محسوب بالمللي ثانية.
152-155نفس الكلام بس للـ refreshToken.
157-159نرجع استجابة HTTP 200 مع جسم الـ response اللي فيه success: true، رسالة، و بيانات التوكنات الجديدة.

5. شرح Class Method: refreshAccessToken()

من ملف src/classes/User.class.ts — الأسطر 435 إلى 504

// السطر 435: توقيع الدالة - async وبتاخد parameters عادي
async refreshAccessToken() {

  // السطر 436: بترجع دالة جوه دالة (Closure)
  // الدالة الجوه دي بتتصرف زي Express Middleware
  return async (req: Request, res: Response, next: NextFunction) => {

    // ============================================================
    // السطر 437-441: استخراج الـ refresh token من الـ request
    // ============================================================
    // بيجرب يجيب التوكن من:
    // 1. cookies (req.cookies.refreshToken)
    // 2. أو Authorization header (Bearer <token>)
    const incomingRefreshToken =
      req.cookies?.refreshToken ||
      (req.headers?.authorization?.startsWith("Bearer") &&
        req.headers?.authorization?.split(" ")[1]);

    // السطر 442-446: لو مفيش توكن خالص → خطأ
    if (!incomingRefreshToken)
      throw new ApiError(
        "Refresh Token is Missing",
        StatusCodes.UNAUTHORIZED,
      );

    // ============================================================
    // السطر 448-451: التحقق من صحة JWT
    // ============================================================
    // jwt.verify بتتأكد من:
    // - التوقيع (signature) سليم باستخدام REFRESH_TOKEN_SECRET
    // - التوكن لسه صالح (مخلصش)
    // لو أي حاجة غلط → throw error
    const payload = (await jwt.verify(
      incomingRefreshToken,
      ENV.REFRESH_TOKEN_SECRET!,
    )) as JwtPayload;

    // ============================================================
    // السطر 453-458: البحث عن المستخدم في قاعدة البيانات
    // ============================================================
    // بنستخدم id اللي جوا الـ JWT payload عشان نلاقي user في DB
    const user = await prisma.user.findFirst({
      where: { id: payload.id },
    });

    // السطر 460-462: لو مفيش user بالـ id ده → خطأ
    if (!user)
      throw new ApiError(`We Couldn't Found User`, StatusCodes.NOT_FOUND);

    // ============================================================
    // السطر 463-468: مقارنة التوكن مع المخزن في DB
    // ============================================================
    // ده أهم سطر في الحماية: بنقارن الـ incomingRefreshToken مع
    // refreshToken المخزن عند المستخدم في قاعدة البيانات.
    // لو مش متطابقين → معناه إن التوكن اتلغى (revoked) أو اتغير
    // مثلاً لو حد سرق التوكن والضحية عملت refresh من قبل كده
    if (user.refreshToken !== incomingRefreshToken)
      throw new ApiError(
        "Refresh Token is Invalid or Has Been Revoked",
        StatusCodes.UNAUTHORIZED,
      );

    // ============================================================
    // السطر 470-477: التحقق من حالة الحساب
    // ============================================================
    // لو المستخدم معمول له Ban → ممنوع يجدد التوكنات
    if (user.isBanned)
      throw new ApiError("Account is Banned", StatusCodes.FORBIDDEN);

    // لو الحساب مقفول مؤقتًا (locked بسبب 5 محاولات فاشلة)
    if (user.lockedUntil && user.lockedUntil > new Date())
      throw new ApiError(
        "Account is Temporarily Locked",
        StatusCodes.FORBIDDEN,
      );

    // ============================================================
    // السطر 479-482: توليد توكنات جديدة (ROTATION)
    // ============================================================
    // بنستخدم id المستخدم كـ payload للتوكن الجديد
    const tokenPayload = { id: user.id };
    const newAccessToken = generateAccessToken(tokenPayload);
    const newRefreshToken = generateRefreshToken(tokenPayload);

    // ============================================================
    // السطر 484-492: حفظ الـ refreshToken الجديد في DB
    // ============================================================
    // بنحدث المستخدم في قاعدة البيانات:
    // - نستبدل refreshToken القديم بالجديد
    // - كده القديم مش هيشتغل تاني لو حد حاول يستخدمه
    await prisma.user.update({
      where: { id: user.id },
      data: { refreshToken: newRefreshToken },
    });

    // ============================================================
    // السطر 494-503: إرجاع النتيجة
    // ============================================================
    return {
      success: true,
      data: {
        accessToken: newAccessToken,
        refreshToken: newRefreshToken,
        accessTokenExpiration: Number(ENV.ACCESS_TOKEN_EXPIRY),
        refreshTokenExpiration: Number(ENV.REFRESH_TOKEN_EXPIRY),
      },
      message: "Access Token Refreshed Successfully",
    };
  };
}

ليه الدالة بترجع Middleware جوه Middleware؟

ده نمط غير تقليدي شوية. الفكرة إن refreshAccessToken() في الـ Class مبتنفذش الحالة مباشرة — بترجع دالة (middleware) جاهزة عشان ننفذها بعدين. السبب: الدالة الجوهية محتاجة توصل لـ req (عشان تجيب الـ cookies), res (عالصفحات الـ cookies), next (للـ error handling). الـ Controller الخارجي عنده access لـ req, res, next فبيمررهم.

ده كان ممكن يتعمل بشكل أبسط: الـ Class method ياخد req, res, next كـ parameters مباشرة. لكن المبرمج اختار النمط ده والسبب مش واضح — ممكن كان بيفكر في reusable middleware pattern.

جدول السطور الأهم:

السطر في الملفالقسمالوظيفة
437-441استخراج التوكنيجيب الـ refresh token من الـ cookies أو Authorization header. req.cookies?.refreshToken — لو مفيش cookie-parser middleware (وهي مش مسجلة!)، هيدي undefined ويروح على الـ header. ده bug مذكور تحت.
448-451jwt.verifyيتأكد من الـ signature باستخدام REFRESH_TOKEN_SECRET (مش ACCESS_TOKEN_SECRET لو استخدم غلط يبقى bug). لو التوكن منتهي أو التوقيع غلط → يحصل error.
464DB comparisonيقارن الـ incoming refresh token مع اللي في قاعدة البيانات. لو غير متطابق — معناه التوكن اتلغى (مثلاً بسبب rotation سابق).
481-482توليد جديديولد accessToken + refreshToken جدد. الـ access token بيطلع بـ ACCESS_TOKEN_SECRET (مدة قصيرة)، والـ refresh بـ REFRESH_TOKEN_SECRET (مدة أطول).
485-492حفظ في DBيخزن الـ refreshToken الجديد في قاعدة البيانات — بكده الـ القديم ملغي. لو حد سرق الـ refresh token القديم، مش هيقدر يستخدمه تاني.

6. دوال توليد التوكنات

من ملف src/helpers/generateTokens.ts

// دالة توليد الـ Access Token:
export const generateAccessToken = (payload: object) => {
  try {
    return jwt.sign(payload, ENV.ACCESS_TOKEN_SECRET!, {
      expiresIn: Number(ENV.ACCESS_TOKEN_EXPIRY),  // 86400000 ms = 24 ساعة
    });
  } catch (error) {
    throw new ApiError("Error generating access token", 500);
  }
};

// دالة توليد الـ Refresh Token:
export const generateRefreshToken = (payload: object) => {
  try {
    return jwt.sign(payload, ENV.REFRESH_TOKEN_SECRET!, {
      expiresIn: Number(ENV.REFRESH_TOKEN_EXPIRY), // 604800000 ms = 7 أيام
    });
  } catch (error) {
    throw new ApiError("Error generating refresh token", 500);
  }
};

7. الـ Middleware: isAuthenticated

من ملف src/middlewares/isAuthenticated.ts

export const isAuthenticated = async (
  req: any,
  res: Response,
  next: NextFunction,
) => {
  try {
    // السطر 13-16: جلب الـ access token من cookie أو header
    const id =
      req.cookies?.accessToken ||
      (req.headers?.authorization?.startsWith("Bearer") &&
        req.headers.authorization.split(" ")[1]);

    // السطر 18-22: لو مفيش توكن → UNAUTHORIZED
    if (!id) {
      return next(
        new ApiError("User not authenticated", StatusCodes.UNAUTHORIZED),
      );
    }

    // السطر 24-28: التحقق من JWT ووضع البيانات في req.user
    if (id) {
      const user = jwt.verify(id, ENV.ACCESS_TOKEN_SECRET!);
      req.user = user;  // الـ controllers اللي بعد كده يقدر يوصل لـ req.user.id
      return next();    // خلّي الطلب يكمل للـ controller
    }
  } catch (error) {
    // لو حصل أي خطأ (توكن منتهي، توقيع غلط، إلخ)
    return next(
      new ApiError("User not authenticated", StatusCodes.UNAUTHORIZED),
    );
  }
};

ازاي الـ Middleware بيشتغل مع الـ Refresh Token Route؟

ملاحظة مهمة: في ملف src/routers/user.router.ts، route الـ refreshAccessToken عليه isAuthenticated middleware. يعنى:

⬇

(يتأكد من صحة access token)
⬇

(يستقبل الـ refresh token من الـ cookies)
🤔 هل ده منطقي؟
الدخول على route تجديد التوكنات محتاج access token سليم؟ فيه جدل:
مع: يضمن إن المستخدم كان مسجل دخول أصلاً.
ضد: لو الـ access token خلص، مش هتقدر تجدد أصلاً!
الحل الأفضل: يبقى route بدون auth middleware، ويعتمد بس على الـ refresh token في الـ cookie.

8. الاعتبارات الأمنية والمشاكل

🐛 مشكلة #1: cookie-parser مش مسجل
في server.ts، مفيش app.use(cookieParser()). ببساطة req.cookies هيكون undefined. يعنى محاولة قراءة req.cookies?.refreshToken هتدي undefined. الحل لازم الكود يشتغل بالـ Authorization header بدل الـ cookies، أو نضيف cookie-parser middleware.
🐛 مشكلة #2: استخراج التوكن من header في دوال مختلفة
دوال refreshAccessToken, logout, verifyRegisterOtp كلها بتجيب التوكن من req.cookies?.accessToken || req.headers?.authorization?.startsWith("Bearer"). لو المستخدم بعت الـ refresh token في الـ header (بدل الـ cookies)، الكود هيقراها كـ access token مش refresh token — ده خلط بينهم.
🐛 مشكلة #3: الدالة refreshAccessToken بتستخدم await مع jwt.verify
await jwt.verify(...) — jwt.verify مش async أصلاً، هو synchronous callback-based. الـ await مش هيضر لكن مش ضروري وبيضلل.
✅ اللي معمول صح:
- استخدام سرين مختلفين للـ access والـ refresh
- مقارنة الـ refresh token مع اللي في قاعدة البيانات (revocation support)
- تدوير (rotation) التوكنات — الـ refresh القديم بينلغى
- التحقق من ban و lock قبل تجديد التوكنات
- httpOnly cookies تمنع XSS من سرقة التوكنات

الخلاصة: تدفق الـ Refresh Token خطوة بخطوة

  1. العميل عنده access token قديم خلص — بيعمل request لـ POST /refresh-accessToken وفي الـ cookies عنده refreshToken.
  2. isAuthenticated middleware — يحاول يتحقق من الـ access token القديم (الموجود في الـ cookies). لو مش موجود أو منتهي → UNAUTHORIZED. (ده ممكن يكون مشكلة لو العميل مش بعت access token قديم).
  3. الـ Controller: new User().refreshAccessToken() ← ترجع middleware function.
  4. تنفيذ الـ middleware: ياخد الـ refresh token من الـ cookies أو الـ Authorization header.
  5. jwt.verify: يتحقق من صحة الـ JWT signature باستخدام REFRESH_TOKEN_SECRET.
  6. يجيب المستخدم من DB باستخدام id الموجود في الـ JWT payload.
  7. يقارن الـ refresh tokens: الجاي من الـ request مع المخزن في DB. لو مش متطابقين → التوكن ملغي أو مسروق.
  8. يتأكد من حالة الحساب: مش banned, مش locked.
  9. يولد توكنات جديدة: accessToken (24 ساعة) + refreshToken (7 أيام).
  10. يحفظ الـ refreshToken الجديد في قاعدة البيانات — بكده القديم ملغي.
  11. الـ Controller يضبط الـ cookies بالتوكنات الجديدة ويرد للعميل.
  12. العميل يستخدم الـ access token الجديد للطلبات الجاية.
📌 نصيحة للتحسين:
1. أضف cookie-parser middleware في server.ts.
2. مدّة الـ access token 15-30 دقيقة أفضل من 24 ساعة.
3. route الـ refreshAccessToken مفروض مايكونش عليه isAuthenticated middleware (أو على الأقل ما يتطلبش access token صحيح).
4. استخدم jwt.verify callback بدل الـ await الزايد.
5. استخدم req.cookies فقط للـ refresh token والـ Authorization header للـ access token — ما تخلطش بينهم.
