🚀 مقدمة عن المشروع

ThreadMarket هو منصة تجارة إلكترونية متعددة البائعين (Multi-Vendor Marketplace) متخصصة في الملابس والأزياء، تستهدف السوق المصري. المنصة بتوصل بين:

  • العلامات التجارية (Brands) - عايزين يعرضوا منتجاتهم
  • المحلات (Retail Shops) - بتشتري جملة
  • العملاء (Customers) - اللي عايزين يشتروا
  • شركات التوصيل (Delivery Partners) - اللي توصل الطلبات

المشروع مبني على Modular Monolith Architecture، يعني إنه تطبيق واحد (monolith) لكن مقسم لموديولات منفصلة جوه الكود نفسه (modules). بعد كده لو احتجنا نوسع، نقدر نفصل كل موديول لـ Microservice مستقل.

٢- الـ Event Bus (للمستقبل): للمهام اللي مش محتاجة رد فوري (زي إرسال إيميل بعد ما الطلب يتأكد):

// shared/event-bus.ts
class EventBus {
  private handlers = new Map<string, EventHandler[]>();

  on(event: string, handler: EventHandler) {
    if (!this.handlers.has(event)) this.handlers.set(event, []);
    this.handlers.get(event)!.push(handler);
  }

  async emit(event: string, payload: any) {
    const handlers = this.handlers.get(event) || [];
    await Promise.all(handlers.map(h => h(payload)));
  }
}
export const eventBus = new EventBus();

🔐 نظام المصادقة (Authentication System)

نظام Auth متكامل مع OTP، JWT، وحماية ضد الهجمات. ده أكمل حاجة في المشروع حاليًا.

التدفق الكامل:

١

التسجيل (Register)

اليوزر يدخل username, email, password + اختياريًا صورة بروفايل. بنعمل hash للـ password بـ bcrypt، بنرفع الصورة لـ Cloudinary، بنولد OTP وبنبعت إيميل للتأكيد. مبنعملش save في قاعدة البيانات لحد ما الـ OTP يتأكد.

٢

تأكيد OTP (Register Verify OTP)

لما يدخل OTP، بنتحقق منه (bcrypt compare) وبعدها بنعمل save للمستخدم في قاعدة البيانات مع isVerified: true.

٣

تسجيل الدخول (Login)

يقدر يدخل بـ email أو phone. بنتحقق من الـ password، لو صح بنولد access token (صلاحيته ٢٤ ساعة) و refresh token (صلاحيته ٧ أيام)، بنحطهم في cookies. لو غلط ٥ مرات، الحساب يتقفل ٣٠ دقيقة.

٤

نسيت كلمة السر (Forget Password)

بنجيله OTP على الإيميل، وبعد ما يدخله صح بنسمحله يغير الباسورد.

٥

تحديث التوكن (Refresh Token)

لما الـ access token يخلص، بنستخدم الـ refresh token (الموجود في الـ cookie) ونطلع واحد جديد و refresh جديد كمان (ده اسمه Token Rotation - بيمنع سرقة التوكنات).

٦

تسجيل الخروج (Logout)

بيمسح الـ refresh token من قاعدة البيانات، ويضبط isOnline: false، ويمسح الـ cookies.

الـ Middleware المستخدمة:

الـ Middlewareالوظيفة
isAuthenticatedيتأكد من صحة JWT الموجود في الـ cookie أو Bearer header و يحط بيانات المستخدم في req.user
isAuthorizedيتأكد إن الـ role بتاع المستخدم مسموح ليه يوصل للـ endpoint ده
validationMiddlewareيطبق Zod validation قبل ما الطلب يوصل للـ controller

ميزات الأمان المطبقة:

  • bcrypt - تشفير كلمات السر (10 salt rounds)
  • JWT بسر منفصل - separate secrets للـ access والـ refresh
  • Token Rotation - كل ما تعمل refresh، بنلغي القديم ونطلع جديد (حماية من theft)
  • Account Lockout - 5 محاولات فاشلة = قفل 30 دقيقة
  • Ban System - تقدير تمنع أي يوزر
  • httpOnly Cookies - منعت الوصول من JavaScript (حماية XSS)
  • Zod Validation - كل المدخلات بيتأكد منها قبل المعالجة
