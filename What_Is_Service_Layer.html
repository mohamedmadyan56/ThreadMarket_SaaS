1. المشكلة اللي بنحلها — ليه أصلاً محتاج Service؟

خليني أشرحلك المشكلة من غير أي كود معقد، بمثال من الواقع:

🍕 تخيل أنك فاتح بيتزا

عندك 3 موظفين:

  • محمد — بياخد الأوردر من الزبون (بيستقبل الطلب)
  • أحمد — بيعمل البيتزا (بيحضر الحاجة)
  • خالد — عنده الفرن والمكونات (الموارد)

دلوقتي، تخيل إن القاعدة بتقول: محمد هو اللي يعمل البيتزا بنفسه!

محمد هياخد الأوردر، يسيب الكاشير، يروح المطبخ، يعجن، يحط الصلصة والجبنة، يشغل الفرن، ويطلع الأوردر. مشكلة: لو ضغط الشغل زاد، مين هياخد الأوردرات التانية؟ مين هيرد على التليفون؟

الحل الصحيح: كل واحد له شغله الخاص. محمد ياخد الأوردر ويوريه لأحمد، أحمد يعمله ويديه لخالد يشويه، خالد يرجعه لأحمد، أحمد يرجعه لمحمد، محمد يقدمه للزبون.

نفس الفكرة بالضبط في البرمجة:

❌

الطريقة الغلط (زي ما هو حاصل في المشروع دلوقتي)

الـ Controller (محمد) مش بس بياخد الطلب — كمان بيعمل Business Logic (يعمل البيتزا)، وبيعمل DB queries (يشغّل الفرن)، وبيعمل Response Logic (يقدمها للزبون). كل حاجة في مكان واحد. أي تغيير يبوظ كل حاجة.

✅

الطريقة الصح (Service Layer)

الـ Controller بياخد الطلب ويوريه للـ Service. الـ Service بيعمل Business Logic وبيستعين بـ Model عشان DB. النتيجة ترجع للـ Controller ويرد على الزبون. كل واحد بيعمل حاجته.

إذن الـ Service Layer هو:

الطبقة اللي فيها "شغل المخ" — القواعد، الحسابات، التحقق من الصلاحيات، اتخاذ القرارات. هو الوسيط اللي بين الـ Controller (اللي بياخد الطلب) والـ Model (اللي بيدخل البيانات).

2. إيه هو الـ Service Layer؟

📖 تعريف أكاديمي

الـ Service Layer هو نمط معماري (Architectural Pattern) بيفصل Business Logic عن طبقة العرض (Controller) وطبقة الوصول للبيانات (Model). بمعنى: هو "المخ" اللي بيفكر ويقرر.

🗣️ تعريف بسيط

هو المكان اللي بنكتب فيه "القواعد" بتاعة التطبيق. زي: "لو المستخدم عنده 5 محاولات فاشلة، اقفله 30 دقيقة". أو "المجموع = السعر × الكمية - الخصم".

🏭 تشبيه المصنع

الـ Controller = استقبال المصنع (يستقبل الطلبات). الـ Service = خط الإنتاج (يعمل المنتج). الـ Model = المخازن (يجيب المواد الخام).

🍕 تشبيه البيتزا

الـ Controller = الكاشير (ياخد الأوردر). الـ Service = الطباخ (يعمل البيتزا). الـ Model = المخزن (يجيب الجبنة والعجين).

إيه اللي بنحطه في الـ Service بالضبط؟

النوعأمثلة
Business Rules"الخصم الأقصى 30%" — "لو المحاولات الفاشلة = 5، اقفل الحساب" — "الشحن مجاني للطلبات فوق 500 جنيه"
Calculationsحساب السعر بعد الخصم — حساب رسوم التوصيل حسب المنطقة — حساب عمولة المنصة
Orchestrationلما تطلب حاجة، Service يستدعي Payment Service + Notification Service + Delivery Service كلهم مع بعض
Validation Logic"مينفعش المستخدم يشتري منتج وهو ممنوع" — "مينفعش البراند يرفع منتج وهو مش verified"
Transformationتحويل البيانات من شكل لشكل قبل ما نرسلها (مثلاً إخفاء الباسورد، إضافة full name من first + last)
📌 تذكر دائمًا: الـ Service بياخد data (string, number, object) ويرجع data (object, array). مش بياخد req, res و مش بيعرف حاجة عن HTTP أو Cookies أو Express.

3. الطبقات الثلاث — Controller, Service, Model

في أي تطبيق Backend احترافي، بنقسم الكود لـ 3 طبقات رئيسية:

🌐 Controller Layer
"إستقبال الطلبات"
⬇
🧠 Service Layer
"Business Logic"
⬇
🗄️ Model Layer (Repository/Data)
"قاعدة البيانات"

🌐 Controller Layer

دوره: يستقبل HTTP Request، يقرا البيانات (body, params, cookies)، يستدعي Service، يضبط Response (cookies, status code, JSON).

مش بيعمل: أي Business Logic، أي حساب، أي Queries.

// Controller — بس استقبال وتحويل
const data = req.body;
const result = await someService.doSomething(data);
res.status(200).json(result);

🧠 Service Layer

دوره: ينفذ Business Logic: قواعد، حسابات، تحقق، تنسيق بين أكتر من Model. بياخد data ويرجع data.

مش بيعمل: مش بيعرف حاجة عن Express, HTTP, Cookies, Response.

// Service — Business Logic بحت
async doSomething(input: string) {
  const user = await model.findUser(input);
  if (user.balance < 100) throw new Error("Insufficient");
  return { allowed: true, user };
}

🗄️ Model Layer (Data Access)

دوره: يتعامل مع قاعدة البيانات. يسوي Queries: Create, Read, Update, Delete. مفيش Business Logic هنا.

مش بيعمل: مش بيتخذ قرارات، مش بيحسب حاجة.

// Model — بس Queries
async findUser(id: string) {
  return prisma.user.findUnique({ where: { id } });
}
🔑 القاعدة الذهبية: كل طبقة تتكلم مع اللي تحتها بس. Controller ← Service ← Model.
الـ Controller مينفعش يروح للـ Model مباشرة.
والـ Service مينفعش يروح للـ Controller.
والـ Model ميعرفش حاجة عن الـ Service أو الـ Controller.

4. مقارنة: من غير Service ← مع Service

خلينا ناخد مثال حقيقي من مشروع ThreadMarket — Login — ونقارن:

❌ من غير Service Layer

الـ Controller بيعمل كل حاجة بنفسه:

// user.controller.ts
export const login = async (req, res) => {
  // 1. Controller بيقرا الـ body ✅ (ده شغله)
  const { email, password } = req.body;

  // 2. ← BAD: Business Logic جوا الـ Controller
  const user = await prisma.user.findFirst({
    where: { email }
  });
  if (!user) return res.status(401).json({...});
  const valid = await bcrypt.compare(password, user.password);
  if (!valid) {
    await prisma.user.update({...});  // ← DB query جوا الـ Controller
    return res.status(401).json({...});
  }

  // 3. ← BAD: DB query جوا الـ Controller
  const token = jwt.sign({ id: user.id }, SECRET);
  await prisma.user.update({
    where: { id: user.id },
    data: { refreshToken: token }
  });

  // 4. Controller بيعمل Response ✅
  res.cookie("token", token, {...});
  return res.json({ success: true });
};

✅ مع Service Layer

كل طبقة بتعمل شغلها:

// 1. auth.controller.ts
export const login = async (req, res) => {
  // ✅ Controller يقرا البيانات من الـ request
  const { email, password } = req.body;

  // ✅ Controller ينادي الـ Service (ويروح يرتاح)
  const result = await authService.login(email, password);

  // ✅ Controller يضبط الـ response
  res.cookie("token", result.token, {...});
  return res.json({ success: true, data: result.user });
};

// ===============

// 2. auth.service.ts
async login(email: string, password: string) {
  // ✅ Service ينادي الـ Model عشان يجيب البيانات
  const user = await authModel.findByEmail(email);
  if (!user) throw new AppError("Invalid", 401);

  // ✅ Service يعمل Business Logic (تحقق من الباسورد)
  const valid = await bcrypt.compare(password, user.password);
  if (!valid) {
    await authModel.incrementFailedAttempts(user.id);
    throw new AppError("Invalid", 401);
  }

  // ✅ Service يولد التوكنات ويرجع data
  const token = tokenService.generateRefreshToken({ id: user.id });
  await authModel.updateRefreshToken(user.id, token);
  return { token, user: sanitize(user) };
}

// ===============

// 3. auth.model.ts
async findByEmail(email: string) {
  // ✅ Model بس Query — مفيش Logic
  return prisma.user.findUnique({ where: { email } });
}

5. القواعد الذهبية للـ Service Layer

🚫 القاعدة رقم 1: الـ Service ماياخدش req, res أو أي حاجة من Express 
// ❌ غلط — Service باخد حاجات من Express
async refreshAccessToken(req: Request, res: Response) { ... }

// ✅ صح — Service باخد data عادي
async refreshAccessToken(refreshToken: string) { ... }
🚫 القاعدة رقم 2: الـ Service ميرجعش Response أو يضبط Cookies 
// ❌ غلط — Service بيعمل res.cookie
res.cookie("token", token, {...});
return { success: true };

// ✅ صح — Service يرجع data بس والـ Controller يتصرف
return { token, user };
🚫 القاعدة رقم 3: الـ Service مش بيعمل DB Queries مباشرة 
// ❌ غلط — Service بيعمل Prisma query مباشرة
const user = await prisma.user.findUnique({...});

// ✅ صح — Service ينادي Model
const user = await authModel.findById(id);
🚫 القاعدة رقم 4: الـ Controller ميناديش Model مباشرة 
// ❌ غلط — Controller بيعمل Query
const user = await prisma.user.findUnique({...});

// ✅ صح — Controller ينادي Service وهو يتصرف
const result = await authService.login(email, password);
✅ القاعدة رقم 5 (إيجابية): Service بياخد primitive types أو objects ويرجع objects 
// Service بياخد حاجات بسيطة ويرجع حاجة واحدة (object)
async login(email: string, password: string): Promise<LoginResult>

async placeOrder(userId: string, items: CartItem[]): Promise<Order>

async updateProfile(userId: string, data: UpdateProfileDto): Promise<User>

6. ليه كل الشركات بتستخدم الـ Service Layer؟

1

🧪 سهولة الاختبار (Testability)

تقدر تختبر Service من غير ما تفتح Express server خالص. مجرد تستدعي الدالة وتتأكد من الـ return value. مثلاً:
const result = await authService.login("test@test.com", "123456");
expect(result.user.email).toBe("test@test.com");
من غير ما تحتاج تعمل HTTP request حقيقي أو تفتح قاعدة بيانات حقيقية (تقدر تستخدم mock).

2

🔧 سهولة الصيانة (Maintainability)

عايز تغير طريقة حساب الخصم؟ روح لـ Service واحد. عايز تغير قاعدة البيانات من PostgreSQL لـ MongoDB؟ روح لـ Model واحد. عايز تضيف Middleware جديد للـ Auth؟ روح لـ Middleware file. مش هتدور على الكود في 10 ملفات مختلفة.

3

👥 العمل الجماعي (Team Work)

في فريق كبير: مبرمج يشتغل على Controller، وآخر على Service، وثالث على Model. مايضربوش في بعض لأن كل واحد في ملفات مختلفة وطبقة مختلفة. ولو حصل conflict في الـ git، سهل تحله.

4

🔄 إعادة الاستخدام (Reusability)

نفس الـ Service ممكن يستخدمه أكتر من Controller. مثلاً authService.login() ممكن تستخدمه في Controller الويب و Controller الموبايل و Controller الـ Admin Panel. نفس الكود، 3 طرق مختلفة.

5

🧠 وضوح Business Logic

لما المبرمج الجديد يفتح المشروع ويدخل على auth.service.ts، يقرا الكود بسرعة ويفهم "قواعد النظام إيه بالظبط". مش محتاج يفتح 5 ملفات ويدور في حاجات HTTP و Cookies و Queries.

6

📈 سهولة التوسع (Scalability)

لما المشروع يكبر وعايزين نفصل Auth Module كـ Microservice مستقل، بناخد فولدر modules/auth/ كله (Service + Model + Controllers) ونحطه في سيرفر لوحده. Service جاهز — بس نضيف له HTTP layer.

7. امتى ما نحتاجش Service Layer؟

مش كل مشروع صغير محتاج Service Layer. في الحالات دي ممكن تستغنى عنه:

الحالةمثالإيه الحل
CRUD بحت API بسيط بياخد data ويحطها في قاعدة البيانات من غير أي منطق (مجرد Create/Read/Update/Delete) ممكن Controller ينادي Model مباشرة. Service هيكون فارغ — مجرد "تمرير" للبيانات.
Prototype / MVP مشروع صغير عاوز تشغله في يومين عشان تجرب فكرة ممكن تبدأ من غير Service Layer وتضيفه بعدين لما المشروع يكبر. (Refactoring سهل لو الكود نظيف).
مفيش Business Logic API مجرد "واجهة" لقاعدة البيانات (زي Firebase أو Supabase مباشرة) استخدم الـ SDK مباشرة — لكن برضه في فرق كبير بين "مفيش Logic" و "Logic بسيط".
⚠️ الخلاصة: لو مشروعك هيشتغل عليه أكتر من شخص، أو هيكون ليه عمر أطول من شهر، أو فيه Business Logic (حسابات، قواعد، صلاحيات) — فأنت محتاج Service Layer. حتى لو المشروع صغير، الفصل العماري ده هينقذك بعدين.

8. الخلاصة — في 5 نقاط

🧠 الـ Service Layer

1. هو الطبقة اللي فيها Business Logic — "مخ" التطبيق.
2. بياخد data (string, number, object) ويرجع data — مالهوش دعوة بـ Express.
3. بين الـ Controller (HTTP) والـ Model (Database) — وسيط محايد.
4. كل شركات البرمجيات بتستخدمه عشان Testing، صيانة، ووضوح.
5. لو مشروعك فيه rules + logic + هيشتغل عليه فريق = Service Layer مش اختيار، هو ضرورة.

المقارنة بوضوح:

✅ دا Service حقيقي

class AuthService {
  async refreshAccessToken(
    incomingRefreshToken: string
  ) {
    // بياخد data (string)
    // ويرجع data (object)
    const payload = jwt.verify(token, SECRET);
    const user = await authModel.findById(payload.id);
    if (user.refreshToken !== incomingRefreshToken)
      throw error;
    const newToken = generateRefreshToken({ id: user.id });
    await authModel.updateRefreshToken(user.id, newToken);
    return { accessToken, refreshToken: newToken };
  }
}

❌ دا مش Service — دا Middleware متخفي في Class

class User {
  async refreshAccessToken() {
    return async (req, res, next) => {
      // ← بترجع function مش data
      const token = req.cookies?.refreshToken;
      // ← بتقرا req (مش شغلها)
      // logic ...
      return { accessToken, refreshToken };
    };
  }
}
الخلاصة: الـ User.class.ts فيه شوية Service حقيقي (زي login()) وشوية Middleware متخفي (زي refreshAccessToken()). هو مش Service Layer نقي — هو "خليط" بين Service و Middleware.
📌 عشان كده في خطة التحويل: أول حاجة هنفصل الدوال اللي بتاخد req, res عن الدوال اللي بتاخد data بس.
login() ← يفضل Service
refreshAccessToken() ← يتعدل ياخد token: string ويرجع data
أي كود بيعمل res.cookie() أو res.clearCookie() ← يتحول للـ Controller
